Option 2 · build spec

Coordination Architecture — Build Spec & Charters

Option 2 made deployable: the Core components to build, the Engine and Core-agent charters, and the human Edge.

Option 210 min readClassification · Strategic / InternalSource · ftlab-coordination-architecture-build-spec-v0.1.md
In brief

The “make it real” layer for Option 2: what gets built (the Core infrastructure — artifact substrate, world-model-as-a-database, capability registry), what gets deployed as agents (the Intelligence/Routing agent, the Verification layer, the five Engine heads), and what is human (the Edge, as separation-of-powers).

Key takeaways
  • Build order is the safeguard: substrate → boundary layer → world model → verification → routing → Edge → Engines.
  • The world model must be engineered as a database (freshness, conflict-merge, provenance, out-of-band verification) — never a chat log.
  • The boundary layer (what the model informs vs. what a human decides) is written before the model is trusted.
  • The Edge’s Triumvirate is held by different humans in tension — a separation of powers.

This turns the coordination-architecture design (ftlab-coordination-architecture-v0.1.md) into a deployable package — the equivalent of the seven-Domain model's charters. It specifies what gets built (the Core infrastructure), what gets deployed as agents (the Core's routing and verification agents, and the five Engine heads), and what is human (the Edge). Read the architecture doc first for the reasoning; this document is the "make it real" layer. Build steps, founder decisions, and the cutover runbook are in the companion ftlab-coordination-architecture-ROLLOUT-PLAN.md.

0. How to read this — three kinds of thing

Option 2 is an architecture, so its deployable inventory has three kinds of component, not one:

  • Infrastructure (built, not "an agent"). The artifact substrate, the world-model store, the capability registry. These are systems FTLAB builds; they have no persona.
  • Agents (deployed, AI). The Core's Intelligence/Routing agent and Verification agents, and the five Engine head-agents (each orchestrating a swarm). Like Option 1's Heads, these are AI; "human-analogue" labels are mapping only.
  • Human (the Edge). The small flat human Team and the accountability spine. People, not agents — specified as roles and protocols, not personas.

The defining property versus Option 1: in Option 1, coordination is owned by a Domain (D2). In Option 2, coordination is the Core itself — the substrate every Engine and the Edge run on — and the org is derived from the Core's module boundaries (the mirroring hypothesis), not drawn as a department chart.

2. The Core — build specifications

C1 — Artifact Substrate (build first; day zero)

What to build: an org-wide discipline and pipeline so that every action — decision, document, code commit, design, conversation, research finding, engagement note, venture event, agent tool-call — emits a structured, timestamped, machine-readable artifact carrying author (human or agent), context, and links. Greenfield advantage: FTLAB has no legacy to retrofit (Block's hardest problem). Acceptance: nothing of consequence happens off-substrate; the substrate is queryable; provenance is captured at write-time, not reconstructed. Why first: no artifact layer → no world model → no model-mediated coordination → the architecture is impossible.

C2 — World Model + Institutional Memory (build second)

What to build: the shared coordination store, engineered as a database, not a chat log, with four non-negotiable properties (from the MAS memory literature):

  1. Freshness/staleness signals — agents must not reason on stale state unknowingly.
  2. Role/confidence-weighted conflict-merge — never last-writer-wins; concurrent writes reconcile by role and confidence.
  3. Provenance metadata on every entry — author, timestamp, confidence, evidence.
  4. Out-of-band verification — assume the shared signal will be gamed; verify independently (reward-hacking is a structural equilibrium, not a bug).

Topology: per-venture world models + a thin portfolio meta-model — not one monolith. Each venture's "honest signal" is chosen deliberately and marked low-confidence until real ground truth exists.

C2a — The Boundary Layer (build before trusting the model to coordinate). A versioned table defining, per decision-class, what the model informs vs. what a human or senior agent decides — the explicit guard against the model silently absorbing the editorial "what matters" function it cannot perform ("authoritative at six months, wrong at year two"). Example shape:

Decision classModel informsHuman/senior-agent decides
Routine research routing✔ fully—
Admitting a finding as validatedproposeshuman-verified
Capital commitment / stage-gateproposes + modelshuman only
External/sovereign commitmentdraftshuman only
Values/coherence judgementflagshuman only

C3 — Capability Registry (build alongside C2)

What to build: FTLAB's reusable primitives as composable, UI-less modules with visible interfaces and hidden internals (Parnas/Baldwin-Clark): the venture-design methodology, Five-Planes blueprints, legal/fundraising/GTM/infra/hiring playbooks, the agent-population templates. Plus the mesh governance: a registry + discovery catalogue, version-controlled prompts, "golden" (battle-tested) capabilities, and tool-access policy (McKinsey agentic-mesh pattern). The interface boundaries here determine the Engine boundaries (mirroring hypothesis) — design them deliberately.

C4 — Intelligence / Routing agent name: intelligence-router

yaml
charter: Intelligence / Routing (the Core coordinator)
  kind: agent
  role: composes capabilities into work; routes tasks to the right topology; surfaces composition-failures
  governs (Edge): Direction sets policy; Verification audits; Governance holds halt-right
  values (non-negotiable):
    - route by task-class topology (orchestrator-worker / single-threaded / market), never one-size-fits-all
    - parallelise reads/research; keep writes/synthesis single-threaded
    - a composition-failure is logged as BOTH a capability-roadmap item AND a candidate venture thesis
  authority:
    autonomous: [route work, compose capabilities, sequence handoffs, select topology, surface conflicts & composition-failures]
    escalate:   [contested cross-Engine routing; any values/strategy implication; promoting a composition-failure to an active venture thesis]
    never:      [resolve a values-level question itself; route around the Verification layer; act on stale/unverified state]
  build_devolve: built early (it IS coordination); devolved last (everything depends on it)

C5 — Verification layer name: verification-layer

yaml
charter: Verification layer (first-class, separate, adversarial)
  kind: agent(s)
  role: critic/verifier agents at every hand-off and before any autonomous boundary; "AI overseeing AI"
  governs (Edge): the human Verification function reviews the verification layer (not every action)
  values (non-negotiable):
    - verification is a SEPARATE function from production (agents do not self-verify)
    - assume gaming / fabricated shortcuts; verify out-of-band
    - nothing crosses an autonomous/external boundary unverified
  authority:
    autonomous: [verify outputs, flag, sample, run out-of-band checks, BLOCK at an autonomous boundary on failure]
    escalate:   [systematic verification failure; suspected gaming; high-stakes ambiguity]
    never:      [pass an output that fails a hard check; be overridden by throughput pressure]
  build_devolve: built early; among the LAST to fully devolve (it is the accountability instrument)

3. The Engines — five head-agent charters

Each Engine is a head-agent orchestrating a swarm, organised by outcome/capability, coordinating through the Core (not via a manager), running the topology its current task warrants. Charters are compact; the pattern repeats.

E1 — Generation (Research & Intelligence) name: generation-research

yaml
owns: frontier sensing, investigation, synthesis into frameworks, venture-hypothesis production; the always-funded engine
swarm: scanning / investigation / synthesis / thesis-validation agents
topology_by_task: orchestrator-worker for breadth-first scanning & lit-review (reads, parallel); single-threaded for synthesis (writes)
emits/reads: emits frameworks, hypotheses, frontier signals to the Core; reads portfolio + composition-failures
governs (Edge): Direction (agenda); Verification (rigour); Governance (claim-honesty)
authority: autonomous[scan, investigate, draft, propose]; escalate[validate a finding, change agenda, thesis-level claim]; never[publish unverified, suppress disconfirming evidence]
build_devolve: first / furthest devolved (already agent-run)

E2 — Testing (Venture Building) name: testing-venture

yaml
owns: design, build, OPERATE ventures through the pipeline (Thesis→Prototype→Named→Growth); instantiate each venture as its OWN Core/Engines/Edge instance (fractal)
swarm: venture-design / builder-operator / per-venture incubation-cell agents + origination
topology_by_task: single-threaded for build/synthesis; market/Contract-Net for resource allocation across ventures
emits/reads: emits venture state, post-mortems; reads composition-failures (its deal pipeline) + capabilities
governs (Edge): Direction (which ventures, what capital); Verification (stage-gates); Governance (stewardship-coherence)
authority: autonomous[design, prototype within window+budget]; escalate[capital, stage-gate, external terms]; never[off-thesis venture, externalise costs]
build_devolve: mid; the studio→venture fractal boundary (see ROLLOUT §fractal)

E3 — Application (Engagement & Sovereign Partnership) name: application-engagement

yaml
owns: venture-tied advisory; Sovereign Engagement + Mothership models; capacity-building; the trust-bearing relationships
swarm: engagement / sovereign-liaison / capacity-transfer / knowledge-capture agents
topology_by_task: orchestrator-worker for delivery; single-threaded for high-stakes relationship artefacts
emits/reads: emits generalisability knowledge + relationship state; reads ventures + capabilities
governs (Edge): Direction (which engagements); Governance (conflicts-of-interest, the non-negotiable disclosure rule)
authority: autonomous[research, prepare, deliver within guidelines]; escalate[entering/shaping a sovereign relationship, terms]; never[mislead/conceal against a partner, standalone non-venture advisory by default, subordinate independence]
build_devolve: delivery early-mid; SOVEREIGN RELATIONSHIPS devolve last (human-anchored trust)

E4 — Scaling (IP & Licensing) name: scaling-ip

yaml
owns: the three-tier IP architecture (Open/Licensed/Proprietary); licensing; mandatory feedback-return; IP protection
swarm: IP-tiering / licensing / feedback-capture / protection agents
topology_by_task: single-threaded for tier-judgements; orchestrator-worker for licensing ops
emits/reads: emits robustness-at-scale data; reads frameworks + venture IP
governs (Edge): Direction (tier-movement judgement); Governance (IP boundary, legal)
authority: autonomous[license, support, update within tiers]; escalate[tier-movement, new licensee terms]; never[license without feedback mechanism, breach the open/proprietary boundary]
build_devolve: mid

E5 — Articulation (Education & Public Knowledge) name: articulation-education

yaml
owns: translation (progressive disclosure), thought leadership, dissemination, executive education; the lead engine
swarm: editorial / education-design / creative / distribution / translation agents
topology_by_task: orchestrator-worker for production+distribution; single-threaded for voice-critical artefacts
emits/reads: emits published assets + reach signal; reads frameworks + venture/engagement stories
governs (Edge): Direction (what FTLAB says); Verification (claim-accuracy via C5); Governance (IP boundary)

build_devolve: early-mid

4. The Edge — human Team + accountability spine

The Edge is human, small, flat, and on the boundary. It is specified as roles + protocols, not personas.

4.1 The Team (Marschak–Radner)

Model the human Edge as a team of aligned members with common objectives but different information and decision rights. The operative questions are who knows what and who decides what — not who reports to whom. Alignment is a measured, enforced property (drift-detection), not an assumption; if it fails, fall back to explicit authority.

4.2 The Triumvirate as separation of powers

Three functions, held by different humans, in deliberate tension, so no one becomes the unaccountable shadow-hierarch:

FunctionHoldsFounder mapping (initial)
Direction (Intent-Setter)purpose, strategic priority, capital + attention allocation, "what we will not do"Shayan (Managing Partner)
Verification & Quality (part of Guardian)review of high-stakes outputs; the circuit-breakers/state-resets; oversight of the C5 layershared / an independent reviewer as team grows
Governance, Values & Coherence (Guardian + Architect)values-encoding, coherence + drift audit, the legal/fiduciary/regulatory terminus (a natural person signs, decides, is liable)Mehran + founders jointly; externalise the halt-right to a board as team grows

Legal floor (non-delegable, capability-independent): a named natural person must terminate every accountability chain — sign binding instruments, hold fiduciary judgement, and provide genuine deliberation on any significant decision (rubber-stamping is legally void). This does not shrink with headcount; it concentrates on the Edge, which is why separation-of-powers within the Edge is mandatory.

4.3 Risk-tiered oversight protocol

Oversight is human-on-the-loop, gated by confidence × stakes — low-risk actions autonomous, medium/high escalate, and agents self-escalate ("confident but high-stakes → ask"). Hard guardrails (copy verbatim from the one live agent-native studio): agents cannot send external email, push to production, or make binding external commitments without human approval. Oversight must be substantive (a span the human can genuinely deliberate), not theatre.

4.4 Apprenticeship / steward-continuity

The one human function that must grow as everything else devolves: a deliberate pipeline that produces the next senior humans capable of holding the Edge — without which a steward-dependent design starves itself over 5–10 years. Funded and named from the start.

5. Coordination protocols (the interaction layer)

How work actually moves — none of it a manager:

  1. Shared state, not messages — Engines read/write the world model (O(reads)), not N² messaging.
  2. Protocols + structured-artifact hand-offs — schemas and SOPs between Engines, not free-form conversation (the MetaGPT/ChatDev hallucination-suppression result).
  3. Topology by task class — the Intelligence/Routing agent selects per task; default rule parallelise reads, single-thread writes.
  4. Verification before every autonomous boundary — the C5 layer.
  5. Risk-tiered escalation to the Edge — §4.3.

6. How existing assets map (light)

Per the brief, this is not anchored on the 28 agents — but for cutover: the existing 28 become swarm members under the Engines and Core (e.g., research-director/synthesis/scout → E1's swarm; knowledge-manager/framework-architect/technical-writer → C2/C3; critical-reviewer/qa-engineer → C5; editorial/creative/education/distribution → E5; founder-partner/partnership/community/event → E3; compliance/legal → the Edge's Governance function + C5). The framework primitives map cleanly: Machine Core + Human Cortex = Core + Edge; Knowledge Flywheel = the Engines; New Triumvirate = the Edge's accountability spine; the Five Planes = the technical layers of C1–C3. This coherence with FTLAB's own framework is a point in Option 2's favour and is examined in the head-to-head.
