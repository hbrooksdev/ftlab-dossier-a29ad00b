Deploy · Option 2

Rollout Plan — Core-first

The turnkey runbook for Option 2: build the Core before deploying any Engine.

Option 24 min readClassification · Strategic / InternalSource · ROLLOUT-PLAN.md
In brief

The Core-first deployment runbook: substrate → boundary layer → world model → verification → routing → Edge → Engines (one venture first) → fractal scale-out, with a verification checklist.

Key takeaways
  • The order is the risk control: build the Core before the org.
  • The boundary layer is written before the model coordinates anything.
  • Deploy Engines on one venture first; instrument decision quality, pre-commit a rollback.
  • Widen autonomy by measured reliability, never by date.

Everything needed to take the coordination architecture (Option 2) live, sequenced so that deployment is mechanical when the founders decide. Nothing in the live system is changed yet. The defining feature versus Option 1's rollout: you build the Core before you deploy any Engine — the substrate first, the org second. Build order here is not cosmetic; it is the safeguard against the architecture's signature failure (a world model that "looks authoritative at six months and is wrong at year two").

2. Decisions already made

  • Coordination is the Core (a built substrate), not a Domain — the org is derived from the Core's module boundaries (mirroring hypothesis).
  • The Edge is a flat human Team; the Triumvirate is a separation of powers held by different humans.
  • Devolution = widen the autonomous tier by measured reliability, not by date.
  • The architecture is fractal — each venture instantiates the same Core/Engines/Edge.

3. Build order (the sequence is the safeguard)

Phase 0 — Foundations (build; deploy nothing yet).

  1. C1 Artifact Substrate — stand up the discipline + pipeline so every human/agent action emits a structured, provenanced artifact. Acceptance: nothing of consequence happens off-substrate; it is queryable. This is the day-zero, highest-leverage build.
  2. C2a Boundary Layer — write and version the decision-class table (model informs vs. human decides) before the model coordinates anything.
  3. C2 World Model — build as a database with freshness, role/confidence conflict-merge, provenance, out-of-band verification; per-venture + a thin portfolio meta-model; choose each venture's "honest signal" and mark it low-confidence until real data exists.
  4. C3 Capability Registry — encode reusable primitives with visible interfaces / hidden internals + the mesh (registry, discovery, golden capabilities, tool-access policy). Design these boundaries deliberately — they determine the Engine boundaries.

Phase 1 — Coordination + verification (deploy the Core agents).

  1. C5 Verification layer — deploy adversarial verifier agents before any Engine is given autonomy.
  2. C4 Intelligence/Routing agent — deploy; wire composition-failure → (capability-roadmap item + candidate venture thesis).

Phase 2 — The Edge (stand up the humans + protocol).

  1. The Triumvirate as separation of powers — assign Direction / Verification / Governance to different humans; write the legal/fiduciary terminus (who signs, who is liable); externalise the halt-right to a board as the team grows.
  2. Risk-tiered oversight protocol — confidence×stakes gating; the hard guardrails (no external email / prod-push / binding external commitment without approval); the apprenticeship/steward-continuity function.

Phase 3 — The Engines (deploy on one venture first).

  1. Deploy the five Engine heads + swarms, organised by the C3 capability boundaries, with model-tiering (heads high-tier, execution agents lighter). Run on one venture first, instrument decision-quality (not just "dashboards green"), pre-commit a rollback criterion, and only then template across the portfolio.

Phase 4 — Fractal scale-out.

  1. Instantiate the same Core/Engines/Edge per new venture; coordinate the portfolio through the thin meta-model + protocols, not a management chain.

4. Verification checklist (before relying on the architecture)

  • Artifact substrate live; coverage audited (what is not emitting artifacts?).
  • Boundary layer written and versioned; every "human-only" decision-class enumerated.
  • World model has freshness + conflict-merge + provenance + out-of-band verification (not last-writer-wins).
  • Verification layer blocks at autonomous boundaries; humans review the verification layer, not every action.
  • Triumvirate functions held by different humans; legal terminus named; oversight spans are substantively reviewable.
  • Composition-failure → venture-thesis wiring works end-to-end.
  • One venture validated through a full cycle before portfolio templating.
  • Cognitive Overhead / Alignment Debt / drift gauges instrumented.

Rollback: the architecture is built incrementally and validated on one venture before scale-out; promote each task-class to autonomy only on measured reliability, and keep human checkpoints until then.

5. Founder decisions (Option 2)

  1. Accept the Core-first reframe — build the coordination substrate and derive the org from it, vs. starting from a department chart.
  2. The "honest signal" per venture stage — what counts as ground truth before real data, and how much the model is allowed to coordinate when ground truth is thin.
  3. The legitimate human floor — which acts are reserved to a named natural person; how separation-of-powers is held within a 2–5 person Edge.
  4. The apprenticeship commitment — fund steward-continuity as an explicit Edge function from day one.
  5. Resolve the "Five Planes" naming collision (technical stack vs org-layers) so architecture and framework stay coherent.
  6. Relationship to Option 1 — adopt Option 2 as primary, run Option 1 first and migrate, or hybridise (the head-to-head analysis informs this).

6. Sequencing note (why Core-first is non-negotiable)

Deploying Engines before the Core exists rebuilds ad-hoc coordination (the thing the architecture removes) and produces the silent year-two failure. The order — substrate → boundary layer → world model → verification → routing → Edge → Engines (one venture) → fractal — is the design's primary risk control. Build it in that order or do not build it.
