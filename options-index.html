This is the index to FTLAB's organisational-architecture options. There are four distinct architectures. Each is documented so it can be reviewed on its own; each has a "design" (what it is + how it deploys) and a "case for it" (the advocate's brief). Read 1, 2, and 4 independently first, then read 3 (the integration) last — it only makes sense once you've seen what it integrates. The full comparison is in the head-to-head.

One-paragraph review brief per option

Option 1 — Seven Domains. Seven AI head-agents — Research & Intelligence; Knowledge, Memory & Coordination (the keystone); Venture Building; Platform & Agentic Systems; Quality & Guardianship; Communication & Ecosystem; Capital, People & Operations — each owning a function with a single-owner charter, under a human Cortex holding Architect / Intent-Setter / Guardian. Strengths: deployable now, regulator-legible, single accountable owner per function, already pressure-tested. Weaknesses: its coordination keystone (D2) was rated "asserted, not engineered"; reads as "AI-enabled" (a C-suite chart with agents in the seats); scales by splitting Domains and adding governors.

Option 2 — Core / Engines / Edge. Coordination is a built substrate (the Core: artifact layer → world model → capability registry → routing + verification agents), not a department. Five Engines (Generation / Testing / Application / Scaling / Articulation) do the Flywheel work and coordinate through the Core; a small flat human Edge holds the undelegatable accountability. Strengths: it is where the research points (the "two jobs of hierarchy" — routing automates, accountability doesn't), genuinely AI-born, fractal answer to governance-at-scale, a built-in deal-origination engine. Weaknesses: gated on a Core-first multi-phase build (long pre-value period), the Core is a single point of failure, accountability specified as functions not named people.

Option 3 — The Integrated. The two above are one design at two altitudes: Option 1's named Domains ≈ Option 2's Engines; Option 1's Cortex ≈ Option 2's Edge; Option 1's D2 ≈ Option 2's Core, minus the engineering. So: keep Option 1's legible named Domains + Cortex as the human-facing accountability face, run Option 2's engineered Core as the coordination engine underneath, reach it by a phased sequence (deploy 1 now, build its keystone to the Core spec, lift coordination into the Core as reliability is measured, migrate Domains→Engines on one venture, scale fractally), and wire in the four Option-4 upgrades. Strengths: captures Option 1's readiness/legibility and Option 2's coordination rigour, pays neither's full penalty, closes Option 1's one Critical finding. Weaknesses: two vocabularies risk confusion (mitigated by one canonical mapping table); a half-built Core is worse than none (mitigated by a hard gate).

Option 4 — The New Mode. Both 1–3 quietly rebuild a central coordinator and design the interior of a firm; the framework actually asked what FTLAB's institutional form is. This mode makes FTLAB a thin constitution (identity + machine-enforced Values + reputation) over an internal market + pressure-field that coordinates a population of agents and venture-cells under selection — with the studio as the genome + selection environment and the whole thing exposed as an open protocol others plug into. Strengths: governance by constituted incentive rather than routing layers; autonomy continuously earned via reputation; prices the scarce resource (attention); treats FTLAB as what it is (a maker of institutions). Weaknesses: internal markets fail under thin signals and opportunism; gaming/collusion are structural; can't clear a market at 3–5 people; the infrastructure is 2026-immature — hence a destination, not a start.

Recommended review order

  1. Option 1 — the familiar, deployable baseline. Read v0.3 (the model), skim domain-charters (the seven specs), then positions/case-for-option-1.
  2. Option 2 — the architecture-first alternative. Read coordination-architecture-v0.1 (the reasoning), skim build-spec (the deployable components), then positions/case-for-option-2.
  3. Option 4 — the challenger / destination. Read positions/case-for-option-4-new-mode (it expands the option space; even if not adopted, it reframes 1–3).
  4. Option 3 — the integration (read last). Read ftlab-org-architecture-option-3-integrated-v0.1 (the design) and positions/case-for-option-3. It only makes sense once you've seen 1, 2, and 4.
  5. The comparison — ftlab-org-architecture-head-to-head-v0.1.md for the full matrix, the four positions side by side, gaps, and the recommendation.

Status

All four options are documented; Options 1–3 are at deployable parity; Option 4 is documented as a destination-stage design. Nothing is deployed live. The set in 02-framework-development/: the four option designs above, the four advocacy briefs in positions/, the head-to-head, plus the supporting artifacts (canonical-terms resolution, venture template, pressure-test, maximal-autonomy end-state).
