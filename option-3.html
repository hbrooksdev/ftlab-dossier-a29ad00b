Option 3 · the path (v0.2)

The Integrated Architecture

The determined path: build the real Core from day one and present it through familiar domain-faces that thin as the Core matures.

Option 317 min readClassification · Strategic / InternalSource · ftlab-org-architecture-option-3-integrated-v0.2.md
In brief

Updated with build-time retired. The prior Critical (“deployable now is a sleight of hand”) hinged on a slow Core build and is withdrawn. Option 3 is now the selected route to the Option 2 destination — one build, with the faces over it from the start.

The faces (the Engines — derived as ~3, not seven) are human-legible accountability labels, not the department model: the Core does the routing, the faces localise accountability.

Key takeaways
  • Build the Core (Ch 5–6 spec) from day one; wear the faces over it — not Option 1 first and Option 2 later.
  • The faces are scaffolding that thins as the Core matures — the load-bearing bet the red-team says must be engineered, not asserted.
  • Two jobs of hierarchy, reconciled: the Core routes, the faces localise accountability — so the faces are not bureaucracy.
  • Carries the inherited Critical: independent Guardianship at 2–5 humans, to be solved first.

Option 3 is no longer one candidate among four. The determination (ftlab-architecture-determination-book1-grounded-v0.1.md §4) selects it as the path FTLAB takes, converging on Option 2 as the destination. This document is therefore a specification of the selected architecture, not a comparative brief. Read the determination first; this document gives the full design, the phasing, and the open criticals.

1. The core idea

Option 1 answers "who is accountable for what?" Option 2 answers "how does coordination physically happen?" An organisation needs both answers, and — critically — they are the same structure described at two altitudes. Combining them is not a compromise but a completion. The mapping is near-exact:

The load-bearing insight: Option 1 lacks an explicit Core. It puts the coordination substrate inside one Domain (D2), as "the integrator + institutional memory" — a keystone that its own v0.1 pressure-test rated the single Critical flaw: "memory-as-coordinator is asserted, not engineered." Option 2's Core is exactly that missing engineering, lifted out of the department layer and built as a real substrate (freshness, role/confidence conflict-merge, provenance, out-of-band verification, a boundary layer). The integration is one move: take Option 1's accountability layer — Domains + Cortex — and re-seat it on Option 2's properly-engineered Core, instead of on the hand-wavy coordination buried in D2.

2. Why Option 3 is the determination's selected path, not a compromise

Retiring the prior Critical

The v0.1 framing pressure-tested Option 3's "deployable now" claim and found it "a sleight of hand": if the Core takes a long time to stand up, customers and partners encounter Option 1's department shell while the Core lags — Option 1 with Option 2 as a distant aspiration, not a genuine integration. That finding was entirely contingent on slow Core build-time being a valid objection.

It is no longer valid. The determination (§8) establishes the governing ground rule for all option re-runs: "the difficulty or duration of building the Core is no longer a valid objection. As long as the Core can be built, a design that depends on it is valid." The same section explicitly retires Option 3's prior Critical: "Option 3's prior Critical that 'deployable now is a sleight of hand' (which hinged entirely on the Core taking a long time to stand up)." With that premise withdrawn, the critique collapses. The Core is built — from the first day, in parallel with standing up the domain-faces — and the sleight-of-hand reading no longer applies.

The affirmative case, grounded in the determination

With the build-time objection retired, the determination §4 reaches its verdict directly: "FTLAB should build the real Machine Core to the Book 1 (Ch 5–6) specification, present its functions through a small set of familiar domain-faces (the Engines), and govern it through a thin Human-Cortex Edge running the New Triumvirate. This is the Integrated path — Option 3 — converging on the Mothership / Core · Engines · Edge destination, Option 2. It is not Option 1 as an endpoint, and not Option 4's internal market."

The determination also explains why the removal of the build-time objection is itself decisive (§4): "Every prior version of this analysis hedged toward Option 1 for one reason only: building the Core was framed as a long, expensive, pre-value undertaking… Retire that premise and the hedge collapses… The only thing Option 1 was ever buying was time-to-first-value during a slow Core build; with that concern removed, Option 1's distinctive advantage disappears and only its distinctive risk — governance theater, calcified departments, routing rebuilt in human hierarchy — remains."

3. The "two jobs of hierarchy" reconciled

A standing concern with any architecture that retains named domain-faces is that it reconstructs the hierarchy it claims to dissolve. The determination (§4) provides the precise reconciliation, and it is worth carrying in full here because it defines the relationship between the Core and the Engines.

Hierarchy does two jobs: it routes information and it localises accountability. Autonomous coordination dissolves the first and not the second. In Option 3:

  • The Core does the routing. No human department sits in the information path. The Core's C4 routing layer handles topology-matched task composition; work does not travel through a Domain's inbox on its way to another Domain. The Domains are not routing nodes.
  • The Engines (domain-faces) localise accountability. "Who owns customer trust? Who owns capital allocation? Who owns venture quality?" These are answered by named functions, each a VP-Agent under a charter, each escalating to a named Cortex role. Accountability is still territorial; routing is not.

The domain-faces of Option 3 are therefore not bureaucracy. They are accountability labels on a substrate that already does the routing. This is the precise reconciliation the framework asks for in §5.3: structure that scales governance without rebuilding the coordination hierarchy. The named faces cost almost nothing to retain and buy three permanent goods that pure Option 2 lacks: external legibility and legal defensibility (a director must be a natural person; a sovereign commitment must have a named accountable human), a stable accountability ontology that does not shift each time the Core's module boundaries are refined, and a human-comprehensible onboarding and governance surface for new Cortex members and external partners.

4. The end-state: the permanent hybrid

This is the structure FTLAB runs on once integrated. The answer to "do the named Domains disappear once the Core matures?" is: no, keep them, permanently, but in a progressively thinner form. Legibility and coordination-efficiency are different requirements with different audiences, satisfied by two artifacts of one structure:

  • Outward / upward (regulators, sovereign partners, board, new hires, the founders' own legal accountability): present the named Domains + human Cortex. "Who owns research? D1. Who can halt on a values breach? The human Guardian. Who is the natural person liable for this sovereign commitment?" — answered by named seats, not abstract functions. GDPR Art. 22 voids rubber-stamp human review; these accountability seats are the structural answer.
  • Inward / operationally (how work actually moves): everything runs through the Core. No Domain routes information to another Domain; they read and write shared state. Topology is task-matched. Verification is a first-class Core layer.

The one permanent structural change: D2 is demoted from "the Domain that coordinates" to "the steward of the Core." Coordination stops being something a Domain does and becomes something the substrate is. This dissolves Option 1's two Critical findings simultaneously — coordination is now engineered (the Core), and is no longer a single-point-of-failure Domain devolved early.

Why keep both faces as the Core matures: the named Domains cost almost nothing to retain once the mapping table is the single source of truth, and they carry governance obligations that the Core cannot discharge — external accountability, legal standing, human comprehensibility. As the Core matures and the domain-faces thin, what thins is the operational workload of the Engines, not the accountability they carry. The faces are scaffolding for human legibility that gradually becomes lighter; they are not the department model of Option 1 used as a permanent routing layer.

5. The path: phases as progressive disclosure, not deferred Core

The prior v0.1 phasing read as "Phase 0 = stand up Option 1; build the Core later." That framing was a product of the now-retired build-time concern, and it is replaced here.

The correct framing (determination §4): the Core and the domain-faces begin simultaneously. Phases describe progressive disclosure — how much of the Core's autonomy is exposed to which task classes, and how Cortex attention is budgeted as the system matures. The Core is not deferred to Phase 1 or Phase 2; it is the substrate from the first day. What phases govern is the widening of the autonomous tier by measured reliability, and the thinning of the domain-faces as the Core demonstrates trustworthiness per task class.

Phase 0 — Foundations: build Core and reveal faces simultaneously.

Stand up the seven Domain accountability labels with their charters, Cortex separation-of-powers (Direction / Verification / Governance held by different humans), and the canonical mapping table. At the same time — not sequentially — begin building the Core to the Book 1 Ch 5–6 specification:

  • C1 artifact substrate, day zero (version-controlled; provenance-tracked)
  • C2a boundary layer first — the per-decision-class "model informs vs. human decides" table, written before the memory is trusted; Agent Charters with hardConstraints prior to reward, in place before any agent is deployed
  • C2 world model as a database (freshness, role/confidence conflict-merge, provenance, out-of-band verification)
  • C3 capability registry
  • Strategy-as-Code repository standing before any autonomous action touches it

D2's first job (unchanged from v0.1): canonical-terms resolution — the ratio stated three ways, the two "Five Planes" definitions, the phantom agents. Cheap proof that the substrate can hold the institution's own terms straight before it coordinates anything else.

Exit trigger for Phase 0: the Core passes Option 2's verification checklist on the studio itself. Until then D2 remains human-curated for contested coordination, and the org runs on Option 1's charters as written. The domain-faces are already in place; it is the autonomous tier that is gated.

Phase 1 — Coordination proven: widen the autonomous tier on measured task classes.

The other Domains stop querying D2 as a department and start reading/writing shared state (O(reads)). Cross-Domain coordination through the Core matches or beats the human-curated path on a measured task class; Cognitive Overhead is flat or falling. Adopt Option 4's four upgrades (§6 below): priced attention, reputation-as-autonomy, values compiled to rules-as-code at the membrane, and a pressure/learning signal on the substrate. Move the Cortex toward full separation-of-powers and set a dated externalisation trigger for the Guardian halt-right.

Phase 2 — Migrate Domains to Engines on one venture first.

Re-cut the Domains into Engines by capability and module boundary. Do it on one venture; instrument decision quality; pre-commit a rollback criterion before migrating. The Engine boundaries evolve to match Core module boundaries; the accountability labels stay fixed. Wire the deal engine: composition-failure surfaces as a capability-roadmap item and a candidate venture thesis.

Phase 3 — Fractal scale-out and measured devolution.

Instantiate Core / Engines / Edge per new venture; coordinate the portfolio through the thin meta-model and protocols, not a chain. Widen the autonomous tier by measured reliability per task class (research/coordination first; synthesis/build; external/irreversible last). The Edge thins to the irreducible legitimacy-and-purpose minimum; apprenticeship is the one Edge function that grows. Named accountability faces remain; operational weight on those faces falls.

6. The four upgrades from Option 4, wired in

The integration is the moment to adopt Option 4's four "steal-ables." None requires the internal market Option 4 depends on; each improves the hybrid at low cost and bends it toward the Option 2 destination.

  1. Price human attention. Give every Engine an attention budget it must spend to escalate to a human. Converts the lagging "Cognitive Overhead Index" metric into a live allocation mechanism — the framework's scarcest resource, priced. (Determination §5: "Cognitive Overhead Index (< 30 = genuinely AI-born) as the diagnostic.")
  2. Reputation as a first-class object. Replace static charters/risk-tiers with a per-agent, per-region reputation and validation registry. Devolution stops being a quarterly founder decision and becomes continuous, earned, and reversible — autonomy rises with verified track record and falls on failure.
  3. Values compiled to rules-as-code at the membrane. Make non-negotiable Values machine-enforced at the protocol boundary, binding regardless of model or vendor — Values-Conscious Architecture as an enforced invariant, not prose. (Determination §6: hardConstraints prior to reward, charter precedes deployment.)
  4. A pressure/learning signal on the substrate. Let the world model drive coordination for decomposable task classes, not just inform it — importing the stigmergy result from Option 4 without adopting the full internal market. Iteration Half-Life as the operating signal; honest exit criteria pre-written; cross-venture learning as the selection-of-ideas mechanism, not internal competition. (Determination §5: reframed steal-able 4.)

These also make the eventual move toward the Option 4 genome form a continuation rather than a rebuild.

7. What it reuses (why it is deployable from day one)

Option 3 orchestrates existing artifacts. The Core is not invented from scratch — its specification is Book 1 Ch 5–6, which is framework §2.2, §3.5, and §5.1, and which the determination calls the build-spec for the Core (§3b). Concretely:

  • From Option 1: the seven staged Domain definitions, the 28→7 elevation map, the canonical-terms resolution, the venture template, the verification-gated rollback. These are the accountability face.
  • From Option 2: the Core build-spec (C1–C5), the five staged Engine definitions, the routing and verification agents, the interaction protocols. These are the coordination engine.
  • From Book 1 Ch 5–6 (via the determination §6): Agent Charters, Strategy-as-Code, the IPRE pipeline, the Governance Loop (Alignment Debt + Risk Twins), override and escalation tiers, the evaluator-agent architecture, statistical oversight and the weekly random audit of 5–10 non-escalated decisions.

New work is integrative and small: one canonical mapping table (Domains ↔ Engines, Cortex ↔ Edge, D2 ↔ Core) as the single source of truth; the four upgrades; and the phase triggers.

8. Assumptions, functioning, and evolution

Assumptions: the Domain ↔ Engine and Cortex ↔ Edge mappings are as clean in practice as on paper (high confidence — explicit in both docs); the Core can be engineered to the Ch 5–6 spec (an engineering programme, not a research gamble — determination §3b; medium confidence, de-risked by proving on the canonical-terms job first); running a legible face over an efficient substrate does not create a gap between represented and real governance (medium-high if the mapping table is the single source of truth and the represented-vs-actual audit is a standing Edge function); founders hold separation-of-powers at 2–5 people or externalise on a dated trigger (medium — a logged, known risk addressed in §9 below).

Likely functioning: seven named, accountable Domains under a small human Cortex — the org chart anyone can read — all of whose coordination happens by reading and writing a shared engineered world model, with verification before every autonomous boundary. Founders govern by policy and exception. New work routes by topology-matched composition; coordination cost scales O(reads); gaps in the Core's coverage surface as candidate venture theses; each venture is the same pattern in miniature.

Evolution: the autonomous tier widens by measured reliability (research/coordination first; external/irreversible last). The named Domains stay fixed (the accountability ontology) while the Core's internals and Engine boundaries evolve underneath — holding stable what must be legible, letting move what is implementation. End-state: a near-fully-autonomous Core and Engines under the smallest legitimate human Edge/Cortex, with apprenticeship the one Edge function that grew — and the four upgrades already bending the substrate toward the Option 4 constitutional-core form.

9. The inherited Critical: Independent Guardianship at 2–5 humans

This is not a risk Option 3 creates. It is inherited by every option and is designated the system's true unsolved Critical by the determination (§7). It is stated here because Option 3 does not solve it, and any reading that implies the integration's elegance addresses governance independence would be false.

The problem: Book 1 Ch 8–9 dissolve the bandwidth half of the oversight problem — evaluator-agents, statistical sampling, the Guardian override-rate as a drift sensor, and the weekly random audit of 5–10 non-escalated decisions make Guardianship a property of the substrate, not just of a person. But the independence half is only partially solved. Ch 9 states the principle: "The Guardian should not carry P&L responsibility… should not report to the CEO." The chapters openly leave the founding-team capture problem: what happens when the Guardian is a founder, and all 2–5 Cortex members share identical financial interests. The Roman-triumvirate warning in the book names it without resolving it. The framework surfaces the same gap in §5.3.

The determination's required answer (§7):

  1. Designate one Cortex member as Guardian with structural protections — no P&L ownership, a documented veto on values/irreversibility/novelty cases, a charter clause the other founders cannot revoke unilaterally.
  2. Build oversight into the Core, not only the person — evaluator-agents, the override-rate sensor, and the weekly random audit make Guardianship a substrate property that survives any single human.
  3. Bring an external check earlier than is comfortable — an independent director or external Guardian-auditor with read access to the audit trail and standing to challenge. At 2–5 Cortex members, genuine independence must come partly from outside.

The determination's instruction (§7): this is the one fix all four options defer, and it should be solved first — before the choice between option-shapes matters, because every shape inherits it.

11. Build steps and founder decisions

Build steps: build Core and domain-faces simultaneously from day one (not sequentially). Follow Option 1's rollout to stand up the seven Domains and their charters, while building the Core to the Ch 5–6 specification in parallel (C1 artifact substrate → C2a boundary layer → C2 world model → C3 capability registry → Strategy-as-Code repository). Wire the four upgrades at Phase 1. Maintain the canonical mapping table from day zero. Phases 1–3 per §5 above.

Founder decisions:

  1. Commit to building the Core to the Ch 5–6 specification from day zero, in parallel with standing up the domain-faces — not after a legibility phase, not deferred to Phase 1. (This is the central bet of Option 3 as the determination selects it.)
  2. Resolve Guardian independence before the design choices matter — structural protections, external check, oversight built into the substrate. (§9 and determination §7.)
  3. Adopt the four Option 4 upgrades now — priced attention and reputation-as-autonomy are design-shaping and should not be retrofitted.
  4. Confirm the permanent hybrid as the end-state — keep the named Domains forever as the accountability face; do not plan to collapse to pure Option 2.
  5. Set a dated externalisation trigger for the Guardian halt-right — not "when we're ready" but a specific trigger condition, logged as Alignment Debt until met.

12. Summary: what makes Option 3 the path

Option 1 ships a Critical unbuilt keystone. Option 2 is the correct destination. Option 3 is the route: build the real Core to specification from the first day, wear the domain-faces over it as accountability labels and human-legibility scaffolding, govern through a thin Cortex running the New Triumvirate, and progressively widen the autonomous tier as reliability is demonstrated. The faces thin as the Core matures; the accountability they carry does not. With the build-time objection retired, Option 3 captures Option 1's legibility and legal defensibility, Option 2's coordination rigour and Mothership architecture, and — via the four upgrades — is already moving toward Option 4's constitutional-core form.

The one line: build Core and face simultaneously, route through the substrate not the Domain, keep accountability territorial, widen autonomy by measured reliability, solve Guardian independence first.
