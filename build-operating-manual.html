Status: Internal / Strategic / Confidential · 2026-06-02 · Phase 3 Compiles against: the org-build foundation (Phase 0), the Core (Phase 1), the three Engines (Phase 2).

How FTLAB runs: a Machine Core, three Engines, and a human Edge. This manual is the day-to-day operating reference; the strategy is in the determination, the spec is in the lexicon, the agents are in the Engine docs.

The Core does the routing; the Engines do the work; the Edge holds accountability. The Engine-faces thin as the Core matures (Option 3 → Option 2).

2. How a piece of work flows (IPRE)

  1. Intent — the Intent-Setter states intent (priorities, constraints, trade-offs) in Strategy-as-Code (core/policies/).
  2. Plan — decomposed into Engine/agent objectives, KPIs, and hardConstraints. This is the last human checkpoint before autonomous execution; origination and capital decisions live here.
  3. Run — the relevant Engine executes via its agents' PRAL loops, within each charter's three decision tiers. The integrator routes hand-offs across Engines through the Core; contested routing escalates to the Edge.
  4. Evaluate — evaluator agents + the oversight scripts surface drift back to the Edge; the Intent-Setter revises intent. Loop.

Worked example: the integrator detects a real need it cannot compose from existing capabilities → logs a candidate venture thesis → routes it to the Venture Engine (Thesis stage, autonomous) → promotion to Prototype is confirmation; any capital commitment escalates to the Edge (Intent-Setter proposes, Guardian guards). Every step writes a decision-trace record.

3. The Edge's cadence

  • Architect (Mehran): owns how the Core and Platform work; approves architecture changes that alter an Engine's authority; governs Platform & Agentic Systems and Coordination & Memory.
  • Intent-Setter (Shayan): sets intent and charters; allocates capital; governs the Venture Engine and Knowledge & Ecosystem.
  • Guardian: continuous — watch the override-rate and Alignment-Debt dashboards (core/scripts/); weekly — run the random audit; on any Veto — a 48-hour post-mortem → charter-revision queue. Carries no P&L; for any decision the Guardian is the founder without the stake in it; the external Guardian-auditor halt-right is brought forward (00-guardian-and-independence-v1.0.md).

4. Adding or changing an agent

  1. Write the charter first (schema: 00-agent-charter-schema-v1.0.md). A charter precedes its agent.
  2. It must compile (the five rules): one accountable owner; hard_constraints.refs resolve in the Values-as-code; the three tiers are total; R3/R4 carry Guardian sign-off; registered in the Core (core/charters/).
  3. Changing the Values-as-code is a values-authorship act by the Edge — never by an agent — version-controlled and Guardian-witnessed.

5. How autonomy widens (the thinning rule)

Autonomy widens by measured reliability, never by date — watched via the override rate, the random audit, evaluator disagreement, and the COI. The widening is symmetric: every gate that widens the Core's autonomy over a class of work withdraws the matching Engine operational mandate, and a failure to thin the face is logged as Alignment Debt that brakes further widening. A named owner holds "face operational weight." (This is the Option-3 red-team's required fix — the faces must be engineered to thin, not assumed to.)

6. Gating preconditions (always in force)

  • The frozen lexicon governs; the Values-as-code hardConstraints are live from the first agent action.
  • R3/R4 (irreversible / regulated / trust-bearing) actions stay at human-escalation until the external Guardian halt-right is live.
  • Origination stays at the Edge (the Intent-Setter); Engines support, the Edge decides.
  • The Core is built and verified before the org depends on it; everything is reversible (Core additive, charters de-registrable, the 28 legacy agents retained until replacements verify).
