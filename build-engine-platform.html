The build · Engine

Engine · Platform & Agentic Systems

The AI-Native Enterprise Framework FTLAB licenses + the agent infrastructure — the Head charter + team, and the Engine’s relationship to the Core it is the outward face of.

26 min readClassification · Strategic / InternalSource · platform-agentic-systems.md
In brief

Builds and licenses the platform the Core runs on (~70% of revenue).

1. Engine Overview

Mandate

Platform & Agentic Systems is the Engine that everything else runs on. Its mandate is to build, maintain, and license the AI-Native Enterprise Framework that FTLAB offers to external clients, and to own the agent infrastructure — the Five Planes technical stack, memory and knowledge systems, integration protocols, and security boundaries — on which every Engine operates. Roughly 70% of FTLAB's projected revenue flows through the licensed platform this Engine produces.

The Engine's authority is bounded by Values (non-negotiable hardConstraints, checked before any reward function) and Preferences (tunable by the Intent-Setter). The Head is a VP-Agent: a Virtual Professional bounded by Values & Preferences, not a management layer.

What this Engine owns

DomainSpecifics
Five Planes technical stackData, Model, Agent, Orchestration, and Actuation planes — design, build, operation, and evolution
Agent infrastructureProvisioning, deployment, observability, and lifecycle management for the full agent population across all Engines
Memory and knowledge systemsInstitutional memory architecture, knowledge graph design, and the retrieval substrate every agent queries
Integration protocolsThe API surfaces, event buses, and hand-off contracts that connect agents, Engines, and external clients
Security and access enforcementIdentity, secrets management, access control, audit trails, and the enforcement layer that makes Values constraints non-bypassable in running code
The licensed platform productThe productised, externally deliverable AI-Native Enterprise Framework, including client onboarding, licensing support, and the feedback loop back into the internal stack
Platform charter registry operationsStoring, versioning, and surfacing the Agent Charter registry on behalf of the Core

What this Engine does not own

  • Values authorship — values are authored exclusively by the human Edge; this Engine implements them in the platform but may not alter them (V4).
  • Capital allocation and partner terms — held by the Edge (Architect, Mehran / Shayan).
  • Coordination decisions and routing logic — the Core's integrator owns routing; this Engine builds the rails the integrator runs on, not the routing decisions made on them.
  • Governance Loop adjudication — the Guardian holds halt-rights and values adjudication; this Engine surfaces violations but does not adjudicate them.
  • Agent creation without a charter — no agent is activated without a complete charter. This is not a preference; it is a platform-level enforcement gate. An agent without a charter may never be deployed.
  • Deploying, modifying, or disabling the Core's Governance Loop — the Governance Loop (Alignment Debt, Risk Twins, decision-tracing) is built on this Engine's infrastructure, but its configuration is owned by the Core and the Edge.

Relationship to the Core

The relationship between this Engine and the Machine Core is the most important boundary in the architecture, and it demands precision.

The Core is the institution's coordination substrate — the memory, the integrator, the charter registry, the IPRE pipeline, the Governance Loop, and the evaluator-agent layer. It routes work. It enforces charters. It carries institutional memory.

This Engine builds and maintains the platform the Core runs on. The Core is, in one direction, the internal instance of the product this Engine produces. When the Core routes a decision, queries the charter registry, or runs an IPRE cycle, it runs on infrastructure this Engine designed and deployed.

This means: the Engine is accountable for the reliability, security, and integrity of the substrate; the Core is accountable for the coordination logic running on it. The two are complementary single-owner contracts. When the Core's integrator routes work, it routes it on rails this Engine built. When this Engine deploys a new agent, the Core's charter registry is the authoritative record. Neither owns both halves.

Practically: this Engine coordinates through the Core (routing, memory, charter registration) and escalates to the Edge (Architect) when a change would alter another Engine's authority, when a security incident occurs, or when a new agent role is required.

How it wires to the Core and Edge

Core interface:

  • All charter registrations flow through the Core's Strategy-as-Code repository.
  • Deployment telemetry and observability data feed the Core's evaluator-agent layer and Alignment Debt monitoring.
  • The IPRE pipeline's Run and Evaluate stages depend on infrastructure this Engine maintains.
  • Coordination requests (routing, hand-offs) flow through the Core's integrator, which runs on this Engine's Orchestration Plane.

Edge interface:

  • Escalation target for this Engine's Head is the Architect (Mehran, holding the Architect function of the New Triumvirate).
  • R3/R4 charters require Guardian approval before registered_in_core: true.
  • Values changes that require platform-level enforcement updates are authored by the Edge and handed to this Engine for implementation — the Engine implements but does not author.
  • Security incidents with potential values-level implications route to both Architect and Guardian.

Decision tiers — plain language

Autonomous (acts and logs): Build, deploy, monitor, and maintain the platform within an approved architecture spec; scale infrastructure; routine security operations; client onboarding under an existing licensing contract; run health checks and integration tests.

Confirmation required (prepares; a peer or the Architect confirms before execution): Any architectural change that touches another Engine's authority boundary or the Core; new agent role creation; security incident response beyond immediate containment; changes to the licensed product's API surface; any action requiring Risk Twins validation.

Human escalation (the Edge decides): Security incident with values-level implications; any action that would weaken a values constraint in the platform; irreversible cross-Engine changes; regulatory ambiguity under DIFC law; unresolved conflict between a client requirement and a hard constraint.

Never (bright lines — no reward overrides these): Deploy an agent without a charter; modify or bypass a values constraint; disable audit or observability; alter the Governance Loop without Edge authorisation; egress data outside permitted boundaries.

3. The Team

The team is an engineering swarm: infrastructure, security, quality, and licensing agents organised around the Five Planes stack. The Head orchestrates. All team agents coordinate through the Core (not through a management chain among themselves). Each carries its own charter; risk classes range R1–R3 depending on the irreversibility of the actions the role takes.

Shared boundary note: qa-engineer and the Core's evaluator layer

The qa-engineer agent performs agent-output validation as part of its quality gate function. This output feeds two consumers: (1) the Platform & Agentic Systems Engine's own quality assurance over the licensed product and the agent population it deploys; and (2) the Core's evaluator-agent layer, which uses the same validation signal to maintain Alignment Debt monitoring across all Engines. The qa-engineer is homed in this Engine (it compiles against the Platform Head's charter) but its agent-output validation output is also a Core input. Changes to the qa-engineer's validation protocol require the Core's evaluator configuration to be updated in step. This boundary must be tracked in the charter registry and in any architectural decision records for either system.

5. Verification — the 5 Compile Rules

Rule 1: One accountable owner. Each agent's role.owns list is disjoint from every other charter's owns list in this document. The one designed overlap — qa-engineer feeding both the Engine's quality layer and the Core's evaluator pipeline — is not an ownership ambiguity; the qa-engineer owns the quality gates and validation signals; the Core's evaluator-agent consumes those signals. The signal interface is documented in §3 and tracked in the charter registry.

Rule 2: hard_constraints resolve. Every refs id in every charter above resolves to a value defined in 00-values-as-code-v1.0.md: V1 (honesty), V2 (permitted activity), V3 (human oversight and halt), V4 (Cortex boundary), V5 (accountability and traceability), V6 (data protection), V7 (stewardship), V8 (fairness), V9 (coherence), V10 (intellectual honesty). No id is referenced that does not exist in that file.

Rule 3: Tiers are total. For each charter, the three decision-rights tiers (autonomous / confirmation-required / human-escalation) collectively cover every action class the mandate implies. The tiers are designed to leave nothing unclassified: where an action class is not explicitly autonomous, it defaults to confirmation-required; where it has irreversibility, novelty, or values implications, it routes to human-escalation.

Rule 4: R3/R4 require Guardian approval. The following agents in this Engine are classified R3: platform-head, agent-platform-engineer, actuation-engineer, security-engineer, security-auditor. Each carries approved_by: guardian in its provenance block. registered_in_core: false on all charters in this document — this is set to true only after founder review and Guardian sign-off.

Rule 5: Registered in the Core. This document is the draft to be registered. On Guardian sign-off, each charter is committed to the Strategy-as-Code repository as a version-controlled file; the charter registry is updated; registered_in_core is set true on each agent; and the .claude/agents/data-engineer.md and .claude/agents/qa-engineer.md files are updated to reflect their new home. Changes thereafter are traced.
