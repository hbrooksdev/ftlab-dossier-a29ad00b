Rollout Plan — Seven Domains

The turnkey runbook for Option 1: build steps, the exact CLAUDE.md edits, migration, verification checklist, rollback.

Option 14 min readClassification · Strategic / InternalSource · ROLLOUT-PLAN.md
In brief

Everything needed to take Option 1 live when the founders decide — sequenced so deployment is mechanical, with a verification checklist and a rollback rule (keep the 28 until the seven verify).

Key takeaways
  • Build D2 first; run the canonical-terms resolution as its first job.
  • Apply the canonical-terms edits to CLAUDE.md; reconcile the phantom agents.
  • Stand up the consolidated rigour gate (D5) and the founders’ Guardian protocol.
  • Elevate the 28 into the seven with model-tiering; add the four new functions.

Everything needed to roll out the seven-Domain organisational architecture is built and saved. This document is the deployment runbook: follow it when the founders decide to go live. Nothing in the live system has been changed yet — the 28 existing agents, the canonical conceptual framework, and the operating CLAUDE.md files are untouched. Deployment is deliberately reversible: the 28 are retired only after the seven are verified.

2. Decisions already made (no re-litigation needed at cutover)

  • Seven Domains now; ten documented as the later end-state to federate into.
  • Every Head and team member is an AI agent; humans are only the thin governance Cortex.
  • Guardian = the two founders jointly, interim (non-driving founder guards each call; logged-COI protocol; externalise to a board/independent reviewer as the team grows).
  • Coordination runs through D2's institutional memory + the integrator (agent), not a management chain.
  • The seven-way publication veto consolidates into D5.

3. Deployment steps (in order)

Step 1 — Build D2 first. Stand up the institutional memory + the integrator (knowledge-memory-coordination). Nothing else coordinates well until the substrate exists. Its first task is Step 2.

Step 2 — Apply the canonical-terms resolution (from ftlab-canonical-terms-resolution-v0.1.md). Exact edits:

  • CLAUDE.md (framework repo) — apply now-safe edits:
    • Five Planes line: change "Five Planes: Strategy, Architecture, Governance, Operations, Intelligence — the organizational layers…" → "Five Planes (technical stack): Data, Model, Agent, Orchestration, Actuation. Organisational units are the seven Domains — see the org architecture."
    • New Triumvirate line: change "Human Judgment + Machine Intelligence + Institutional Design" → "New Triumvirate (governance roles): Architect / Intent-Setter / Guardian." (If the capability-source triad is kept, rename it "the three capability sources.")
    • VP-Agent line: change "Virtual professionals (AI agents)…" → "VP-Agent = a Virtual Professional (the Head/orchestrator agent), bounded by Values & Preferences (V&P) — Values non-negotiable, Preferences tunable, with structural preferences between."
    • 5:100 line: add "(illustrative, non-binding; size each Domain's agent population to its function and sector — see the ratio resolution)."
    • Agent Roles section: mark the six-plane / 28-agent roster superseded by the seven Domains (point to the charters).
  • ftlab-website CLAUDE.md: reconcile the "1:500" ratio statement to the non-binding principle.
  • Conceptual framework → v1.1 (a deliberate, founder-reviewed pass — NOT an inline rush): fold Resolutions 1–4 into the definitions (the framework's §2.2 Five Planes and §2.3 Triumvirate are already canonical; add the VP-Agent dual-naming note to §5.1; add the ratio non-binding note); add a new section "Organisational Architecture" that references v0.3 + the charters. Save as ftlab-conceptual-framework-v1.1.md (preserve v1.0).
  • Phantom agents (Resolution 5): handled in Step 4 (the 28→7 migration removes the dangling "Deal Scout" / "Platform Architect" references).

Step 3 — Stand up D5 + confirm the Guardian protocol in writing. Consolidate the rigour gate into quality-guardianship; write the founders'-joint Guardian conflict-of-interest protocol (which founder guards which class of decision) and log the independence gap as Alignment Debt.

Step 4 — Elevate the 28 into the seven. Copy the seven staged definitions from rollout/agents/ into .claude/agents/. Re-home each of the 28 as a team/sub-agent under its Head per the charters' §1 map; reconcile the phantom "Platform Architect" (→ D4) and "Deal Scout" (→ D3) and the dangling data-engineer/qa-engineer reports-to lines. Introduce model tiering: the seven Heads on the high-capability/orchestrator tier; execution sub-agents on lighter tiers (update model: fields and settings.json).

Step 5 — Add the four new functions. Venture builder/operator (D3), platform engineering + security (D4), finance/capital (D7), apprenticeship (D7).

Step 6 — Update coordination wiring. Revise .claude/agent-teams.md to the seven-Domain teams; confirm settings.json (model tiering, hooks, the terminology gate).

4. Verification checklist (before retiring the 28)

  • Every Domain has exactly one Head agent definition, live.
  • Every one of the 28 is re-homed under a Head or explicitly retired.
  • No phantom references remain ("Deal Scout", "Platform Architect" dangling).
  • D2 memory + integrator are live, and the canonical-terms resolution has been applied.
  • D5 rigour gate is the single publication gate; the founders' Guardian protocol is written.
  • Model tiering is set (Heads high-tier; teams lighter).
  • Cognitive Overhead / Alignment Debt / Iteration Half-Life gauges are instrumented.

Rollback: keep the 28 in place until this checklist passes. Retire them only after the seven are verified in operation.

5. Sequencing note

The order matters: D2 before charters widen (the substrate first), terms resolved before agents rely on the memory (so they share one vocabulary), D5 + Guardian protocol before authority is devolved (oversight before autonomy). Build order is not cosmetic — it is the safeguard.
