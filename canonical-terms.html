Canonical Terms Resolution

Domain 2’s first job — making the institution’s own terms consistent (the ratio, the two “Five Planes,” the phantom agents).

6 min readClassification · Strategic / InternalSource · ftlab-canonical-terms-resolution-v0.1.md
In brief

The inaugural exercise of the coordination keystone: resolve the corpus’s contradictions (the human:agent ratio stated three ways; “Five Planes” defined two ways; the New Triumvirate and VP-Agent dual definitions; phantom agents; leadership titles) into single authoritative decisions.

Key takeaways
  • If the memory can’t keep the institution’s own terms straight, it can’t coordinate anything.
  • Retire any fixed ratio; size each Domain’s agent population by function and sector.
  • “Five Planes” = the technical stack only; the organisational units are Domains.
  • VP-Agent = a Virtual Professional bounded by Values & Preferences — a defined pair, not a clash.

This is the inaugural exercise of Domain 2 (Knowledge, Memory & Coordination). The organisational model has Domains coordinate through a shared institutional memory rather than a management chain — which only works if that memory keeps the institution's own terms straight. Today it does not: the corpus defines several core terms two or three incompatible ways. Resolving them is therefore D2's first task, and the test of whether memory-as-coordination is real. Each resolution below is a proposed canonical decision, to be folded into conceptual framework v1.1 and the operating guides. Nothing canonical has been rewritten yet — see §7 for the application list and what awaits founder confirmation.

Why this matters

If two agents read "the ratio" and one finds 5:100 while another finds 1:500, they will size agent populations 25× apart and neither will be wrong against the corpus. If one reads "Five Planes" as a technical stack and another as organisational layers, they will build incompatible mental models of the same institution. Coordination-through-memory fails silently exactly here. So before any charter is written or any Domain is stood up, the shared memory must hold one authoritative definition per term. The six contradictions below are the ones that bear on organisational design; D2 should continue this audit across the rest of the corpus as standing work.

Resolution 1 — The human-to-agent ratio

The contradiction. Three different figures appear: 5:100 (business plan, framework CLAUDE.md, whitepapers), 1:500 / 5:2,500 (website CLAUDE.md), and 1:100 to 1:10,000, sector-dependent (the AI-Born book).

Resolution. No fixed ratio is load-bearing. The operative principle is: a small human Cortex (single digits to low tens) governs a large agent population, with each Domain's agent population sized to its function by risk, reversibility, and regulatory intensity — tighter (more human oversight, more escalation) in regulated/trust-bearing functions; larger and more autonomous where actions are low-stakes and reversible. "5:100" may be kept only as an external, illustrative shorthand, explicitly flagged as evocative rather than binding.

Why. The founders themselves state the exact numbers are unclear and that FTLAB will "always maintain a ratio of a small human team to a large agentic workforce" — a principle, not a number. Fixing a number invites false precision and the wrong sizing math.

Apply to: framework v1.1, both CLAUDE.md files, the 5:100 whitepaper (add the non-binding caveat), v0.3 §9.

Resolution 2 — "Five Planes"

The contradiction. Used two incompatible ways. Framework v1: the technical stack — Data, Model, Agent, Orchestration, Actuation. CLAUDE.md: organisational layers — Strategy, Architecture, Governance, Operations, Intelligence. (A third, separate "six org-planes" taxonomy groups the 28 agents — Intelligence/Strategy/Editorial/Governance & Ops/Community/Culture.)

Resolution. "Five Planes" means the technical stack only (Data/Model/Agent/Orchestration/Actuation) — this is the canonical, framework-grounded definition. The organisational units are Domains (the seven). The CLAUDE.md "organisational layers" use of "Five Planes" is retired, and the "six org-planes" grouping of the 28 agents is superseded by the seven Domains.

Why. One word, one meaning. The collision between a tech-stack and an org-layer sense of "Five Planes" is precisely the kind of ambiguity that breaks coordination.

Apply to: CLAUDE.md (rewrite the "Five Planes = Strategy/Architecture/…" line to point to Domains; reserve Five Planes for the stack), the agent roster's plane grouping, v0.3.

Resolution 3 — "The New Triumvirate"

The contradiction. Framework v1: three governance roles — Architect, Intent-Setter, Guardian. CLAUDE.md: a different triad — Human Judgment, Machine Intelligence, Institutional Design.

Resolution. "The New Triumvirate" = Architect / Intent-Setter / Guardian (governance functions) — canonical, and the one with developed decision-rights. The "Human Judgment / Machine Intelligence / Institutional Design" triad is a different concept (sources of capability, not governance roles); if retained at all, it must be renamed (e.g., "the three capability sources") so it does not collide with the Triumvirate.

Why. The governance model is load-bearing for the whole org design; it needs one unambiguous definition.

Apply to: CLAUDE.md, framework v1.1 (confirm the canonical use).

Resolution 4 — "VP-Agent"

The contradiction. "VP" is expanded two ways: Values and Preferences (framework v1 — the governance-encoding model) and Virtual Professional (CLAUDE.md / whitepaper — the agent-as-institutional-participant).

Resolution. These are complementary, not competing — and should be named as a defined pair, not a collision. A VP-Agent is a Virtual Professional — the orchestrator-level "Head" agent that owns a function. It is bounded by Values and Preferences (V&P) — Values are non-negotiable constraints, Preferences are tunable guidelines, with a third tier of structural preferences between them. So: Virtual Professional names the role; Values & Preferences names how its authority is bounded. Always introduce both together to prevent the apparent clash.

Why. Both ideas are useful and both are in active use; the only problem is the unexplained shared initials. Naming the pairing explicitly resolves it.

Apply to: framework v1.1, CLAUDE.md, the VP-Agent whitepaper.

Resolution 5 — Phantom agents in the roster

The contradiction. The agent registry references roles that do not exist as definitions: "Deal Scout" and "Platform Architect" (in agent-orchestrator.md and agent-teams.md), and data-engineer.md and qa-engineer.md report to a non-existent "Platform Architect."

Resolution. Remove the phantom references. Their intended functions now have clear homes in the seven Domains: "Platform Architect" → Domain 4 (Platform & Agentic Systems); "Deal Scout" → split between Domain 3 (Venture Building) and Domain 1 (Research & Intelligence, scouting). The dangling "reports to Platform Architect" lines re-point to Domain 4.

Why. Broken references propagate into any redesign built on top of them; the elevation to seven Domains is the moment to clear them.

Apply to: agent-orchestrator.md, agent-teams.md, data-engineer.md, qa-engineer.md (handled as part of the 28→7 elevation in the charters document).

Resolution 6 — Leadership titles

The contradiction. Business plan: Co-CEO / Co-CEO. Founder responses: Managing Partner (Shayan) / General Partner (Mehran).

Resolution. Managing Partner / General Partner internally — "CEO" connotes a command hierarchy the model rejects; "Partner" plus a named Triumvirate function fits the governance model. Retain CEO-equivalent titles only where external or DIFC legibility requires them.

Why. Internal language should match the actual governance structure (founders govern agent Domains via the Triumvirate; they do not sit atop a management chain).

Apply to: internal docs and framework; external/DIFC materials may keep CEO-equivalents.
